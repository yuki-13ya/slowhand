# AGENTS.md

このリポジトリでコード生成・編集を行うAIエージェント（Claude Code, Codex等）向けの作業方針です。人間の開発者にも共通して適用されます。

## 基本方針

**「小さい部屋を積み重ねて1軒の家を建てる」ようなやり方はしない。**

- ページ単位・コンポーネント単位で局所最適な修正を積み重ねるのではなく、常に **サイト全体を俯瞰した上で** 個々の変更を位置づけること
- 見た目は「人の目から見てシンプルできれい」を基準にする。装飾や例外ルールを増やす方向ではなく、削って揃える方向で判断する
- 新しいCSSルールを書く前に、同じ目的・同じ色・同じコンポーネントの既存ルールが `style.css` 内に無いか確認する。あれば流用・統一し、似て非なる値を新規に増やさない

## 実装前の宣言・承認

**コード（style.css・functions.php等）を変更する作業に入る前は、必ず「これから何をするか」を宣言し、ユーザーから明示的なOKをもらってから着手する。**

- 要件がまだ確定していない、またはすり合わせの途中である場合は、実装せずに要件を確認する
- 「変更してほしい」という発言があっても、それが具体的な実装指示なのか、まず相談・すり合わせをしたいだけなのかを都度確認する
- 宣言には「何を」「どのファイルを」「どういう方針で」変更するかを含め、ユーザーが中身を判断できる粒度にする
- 承認を得られるまでは、調査・確認・提案（テキストでの説明、色見本の提示など）に留める
- 宣言した項目が複数ある場合、ユーザーがそのうち一部だけに回答したら、**残りの項目には回答がまだ無い**ものとして扱う。一部への回答を全体への許可と解釈しない
- 明確に「進めて」「OK」等の許可が出るまでは着手しない。曖昧な相槌や別の話題への移行を許可とみなさない

## 具体的な作業ルール

### CSS / デザイン

- 色・余白などの繰り返し使う値は、可能な限り [CSS カスタムプロパティ](https://developer.mozilla.org/ja/docs/Web/CSS/Using_CSS_custom_properties)（`:root` の `--color-primary` 等）で管理し、直書きの16進カラーコードを新規に増やさない
- 既存の直書きカラーを触る際は、それが「意図的な別色」なのか「本来同じ色のつもりが少しズレたもの」なのかを判断してから統一する（無条件に一括置換しない）
- 1つの修正が他ページ・他ブレークポイント（PC/タブレット/スマホ）にどう影響するかを確認してから確定する。このサイトはPC/スマホでルールが分かれている箇所が多い
- `style.css` にセクションを追加する際は、既存のコメント区切り（`/*** ... ***/`）の流儀に合わせ、どこに何があるか後から追いやすい状態を維持する

### 変更の粒度

- 一度に大量の見た目変更を混ぜない。「色を変える」「レイアウトを直す」など目的ごとに変更を分け、都度確認できるようにする
- 本番はFTP/SFTP手動アップロードのため、ローカルでの変更後は差分を明確にしてから反映する（[README.md](README.md) のデプロイ手順を参照）

### リポジトリのスコープ

- git管理対象は `wp-content/themes/lightning-child` とドキュメント類のみ。WordPressコア・親テーマ・プラグイン・アップロードファイルは対象外（詳細は [README.md](README.md)）
- コア・親テーマ・プラグインのコードは編集しない。カスタマイズは常に子テーマ側で行う

## 進め方

作業前に必ず以下を確認する:

1. 今回の変更が既存のどの部分（色・コンポーネント・ページ）に影響するか俯瞰する
2. 既存コードに同種の実装がないか探す（ゼロから足さない）
3. 変更後、対象ページ以外に意図しない影響が出ていないか確認する

大きめの変更（CSS全体の整理、色の全面変更など）に着手する前に、作業範囲と方針をユーザーと合意してから進める。
